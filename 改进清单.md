# 改进清单

## 一、缺失的关键实现

### 1. 真正的新闻正文抓取（Step 3 遗留）
- **现状**：只存了 RSS 的 `description`，`content` 一直为 `NULL`。
- **需要**：在采集器中增加 `fetch_full_content` 函数，从新闻链接抓取 HTML 并提取正文。可以使用 `scraper` 或 `reqwest` + 简单规则（如提取 `<article>` 或所有 `<p>`）。
- **影响**：详情页没有完整内容，用户体验差。

### 2. Yahoo Finance API 实际调用（Step 14 遗留）
- **现状**：`price_change` 判定器只模拟返回 `true`。
- **需要**：实现真正的价格查询。Yahoo Finance 需要构造 URL，处理日期和 symbol。可以使用 `reqwest` 获取 JSON 并解析收盘价。
- **影响**：价格类预测无法自动判定。

### 3. 央行利率 / 经济数据的真实来源
- **现状**：`central_bank` 和 `economic_data` 判定器只打印日志，没有对接真实数据 API。
- **需要**：可以对接央行官网、FRED API 或手动录入（因为免费 API 有限）。学习阶段可以模拟，但文档应说明。

### 4. LLM 真实调用（Step 9 遗留）
- **现状**：`call_llm` 函数只返回模拟文本。
- **需要**：接入 Gemini / OpenAI API（需要 API Key），处理请求和熔断弃权。
- **影响**：人类与 LLM 的对照预测无法真正产生对比价值。

### 5. 用户认证与权限
- **现状**：所有 API 固定使用 `user_id = 1`。
- **需要**：增加登录/注册功能，使用 JWT 或 Session，中间件提取 `user_id` 注入 handler。
- **影响**：多用户无法区分，管理员权限无法控制。

### 6. 冷热分离 Worker（Step 12）未实现？
- **检查**：对话中 Step 12 的代码是否写入了？之前有提供 `start_cleanup_worker` 但可能在 main 中未启动。
- **需要**：确认 `src/cold_storage.rs` 存在且在 `main` 中启动定时清理 `content`。

### 7. 动态噪声词更新（Step 3 部分）
- **现状**：`jieba-rs` 已添加，但没有实现每小时扫描 `noise_flags` 并更新动态排除词库的逻辑。
- **需要**：在采集器调度器中增加一个额外任务，定期统计高频噪声词，更新 `Arc<RwLock<HashSet<String>>>` 并在采集过滤时使用。

## 二、代码层面的修改点

### 1. 错误处理统一
- **现状**：混合使用 `Box<dyn Error>`、`unwrap`、`expect`。
- **建议**：为每个模块定义自己的错误枚举（实现 `std::error::Error`），避免 `unwrap`，至少用 `?` 传播。

### 2. 配置外置
- **现状**：各种常量硬编码（如 LLM 熔断阈值 3 天、正文长度 100、用户固定 ID、端口 8000）。
- **建议**：使用 `config.toml` 或环境变量统一配置，读取到全局结构体。

### 3. 日志改进
- **现状**：部分关键操作缺少 info/error 日志。
- **建议**：增加采集、判定、LLM 生成等步骤的成功/失败计数日志，便于监控。

### 4. 数据库连接池大小
- **现状**：`max_connections=5`，对于采集+API+Worker 可能不够。
- **建议**：提高到 20-50，或根据 `DATABASE_URL` 配置动态设置。

### 5. 部分索引可能需要调整
- 检查 `idx_predictions_llm_queue_scan` 是否真正被 `FOR UPDATE SKIP LOCKED` 查询利用。可能需要包含 `user_id` 或 `submitted_at`。
- 建议使用 `EXPLAIN ANALYZE` 测试查询性能。

## 三、前端完善

### 1. 导航菜单
- **现状**：没有菜单链接到统计页和管理员页。
- **建议**：在 `index.html` 顶部添加导航栏，链接到 `/stats.html` 和 `/admin.html`（仅管理员可见）。

### 2. 推演表单提交后的反馈
- **现状**：提交后只弹出 alert。
- **建议**：显示成功消息，并清空表单或折叠。

### 3. 日期选择器的时区问题
- **现状**：前端用 `new Date()` 获取客户端时区，可能导致预期日计算偏差。
- **建议**：统一从后端获取当前 UTC 日期，或者要求用户只选未来日期（前端强制最小日期为服务器返回的 `UTC+1`）。

### 4. 管理员页面权限控制
- **现状**：任何人可访问 `/admin.html`。
- **建议**：前端也做登录检查，未登录或非管理员重定向。

## 四、缺失的文档和测试

### 1. 单元测试
- **现状**：几乎无测试。
- **建议**：为 `brier_score`、`asymmetry_ratio`、`parse_rss_pub_date` 等纯函数编写单元测试。

### 2. API 文档
- **现状**：没有自动生成文档。
- **建议**：使用 `utoipa` 为 axum 生成 Swagger 文档。

### 3. 部署脚本
- **现状**：只能手动 `cargo run`。
- **建议**：提供 `Dockerfile` 和 `docker-compose.yml`（含 PostgreSQL）。

---

## 五、性能与安全隐患

### 1. 采集器无超时控制
- 某些 RSS 源可能卡死，应设置 `reqwest::Client` 的超时（如 30 秒）。

### 2. SQL 注入风险
- 使用 `sqlx` 的 query macro 已编译时检查，安全。

### 3. 密码哈希
- `users` 表中存储明文密码哈希？目前没有注册功能，哈希字段为 `dummy_hash`。真实应用需使用 `argon2` 或 `bcrypt`。

### 4. 无请求限流
- API 可能被滥用，可添加 `tower_governor` 限流中间件。

---

## 总结优先修改清单

按重要性排序：

1. **实现真正的 Yahoo Finance API 调用**（否则价格预测判定无效）。
2. **增加用户认证**（至少登录固定用户，避免硬编码 user_id）。
3. **完成冷热分离 Worker**（释放存储空间）。
4. **错误处理统一**（去掉 unwrap，增加有意义的错误日志）。
5. **实现动态噪声词更新**（让噪声标记闭环生效）。
6. **完善前端导航和时区处理**。
7. **补充部署文档和测试**。
